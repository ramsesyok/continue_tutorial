# エージェント向け チュートリアル執筆指示書

本書は、Continue.dev VS Code 拡張のオンプレミス・エアギャップ環境向けチュートリアルを、章単位で執筆するエージェントに向けた共通指示書です。**新しい章の執筆を開始する前に、必ず本書を最初から最後まで読んでから作業に着手してください。**

---

## 1. プロジェクト概要

本プロジェクトは、[Continue.dev](https://www.continue.dev/) の VS Code 拡張を、完全なオンプレミス・エアギャップ環境でローカル LLM に接続して利用するための、ソフトウェア開発者向けチュートリアルを作成するものです。

- **想定読者**: AI コーディング初心者のソフトウェア開発者（GitHub Copilot 等の利用経験を前提としない）
- **対象 IDE**: VS Code のみ（JetBrains 等は対象外）
- **環境**: 完全オンプレミス・エアギャップ環境
- **LLM**: ローカル LLM。Continue からの接続設定は完了している前提
- **出力形式**: Markdown（[MkDocs](https://www.mkdocs.org/) でビルド、[Foam](https://github.com/foambubble/foam) 互換）

## 3. 執筆ルール

### 3.1 ファイル形式と配置

- 形式: Markdown（`.md`）
- 文書システム: MkDocs（テーマは Material for MkDocs を想定）
- Foam 互換のリポジトリ構造とする
- 配置先: `docs/` 配下
- ファイル名: `NN-slug.md` 形式（例: `01-introduction.md`、`07-autocomplete.md`）
- 付録は `docs/appendix/` 配下に配置（例: `docs/appendix/config-reference.md`）
- 画像は `docs/images/<章番号 2 桁>/` 配下に配置

### 3.2 Frontmatter（Foam 互換）

各 Markdown ファイルの先頭に、YAML 形式の Frontmatter を必ず付与してください。**`title` / `type` / `tags` の 3 つは必須項目です。**

```yaml
---
title: "第N章 章タイトル"
type: chapter
tags:
  - continue
  - vscode
  - on-premise
  - air-gap
  - <章固有のタグ>
---
```

- `title`: 章番号と章タイトルを含む人間可読の文字列。コロンを含むためダブルクォートで囲む
- `type`: 文書種別を表す。次のいずれかを使用する
  - `chapter`: 本編の章
  - `appendix`: 付録
  - `index`: トップページ・索引
  - `meta`: 本書のようなメタ文書
- `tags`: 共通タグ（`continue`、`vscode`、`on-premise`、`air-gap`）に加え、章固有のタグ（`autocomplete`、`agent`、`security` 等）を 1 件以上追加する

### 3.3 文章スタイル

- 言語: 日本語
- 文体: 「です・ます」調で統一
- 想定読者は AI コーディング初心者。専門用語には **初出時に簡単な説明を添える**
- 章の冒頭に「**この章で学ぶこと**」、末尾に「**まとめ**」と「**次の章へ**」を必ず配置する
- 段落は適度に区切り、1 文 80 文字程度を目安に過度に長くしない
- 半角英数字と日本語の間にはスペースを 1 つ挿入する（例: 「VS Code を起動して」）

### 3.4 Markdown 記法

- 本文の見出しは `##`（H2）から開始する（H1 はページタイトル＝ Frontmatter の `title` で代替）
- 見出しレベルは飛ばさず順に使う（`##` → `###` → `####`）
- コードブロックには言語識別子を必ず付与する（` ```yaml ` / ` ```bash ` / ` ```python ` 等）
- 強調は `**太字**`、注意喚起は MkDocs Material の admonition（`!!! note` / `!!! warning` / `!!! tip`）を利用する
- 章間リンクは MkDocs 標準の相対パス（`[第 2 章](02-prerequisites.md)`）と、Foam の wikilink 形式（`[[02-prerequisites]]`）の両方が機能するよう、相対パス形式を基本としつつ wikilink も併用可とする
- 画像参照は `![代替テキスト](images/07/screenshot-01.png)` の形式

### 3.5 サンプルコード・設定値

- ローカル LLM の例として、Ollama / vLLM / LM Studio のいずれかを各章の文脈に応じて使い分ける
- 実在の社内サーバ名・URL・認証情報は記載しない。プレースホルダ（`<your-internal-llm-host>`、`<your-model-name>` 等）を使用する
- `config.yaml` の例は**最小限で有効な構成**を提示する。冗長な設定は避ける
- コマンド例には、コピペで動く形式と、各オプションの意味の説明をセットで記載する

### 3.6 エアギャップ前提の徹底

これは本チュートリアル最大の特徴です。以下を厳守してください。

- 外部 API・外部サーバへの接続を前提とする手順を含めない
- クラウド埋め込みモデル等、エアギャップで利用できない機能に言及する場合は「**本構成では使用しません**」と明示する
- 外部 URL は本文の「手順」中では使わない。「参考リンク」セクション（章末）に分離する
- 「インターネットに接続して〇〇する」「`pip install ...` でダウンロードする」等の表現は、必ず**事前にミラー／オフライン配布した前提**で書き直す

## 5. 各章執筆の標準フロー

1. **本指示書を読む**（必須）
2. **対象章の節構成案**（H2／H3 レベルの見出し一覧と、各節のねらいを 1〜2 行で）を作成し、ユーザーに提示して承認を得る
3. 承認後、本文を執筆する
4. **執筆後にセルフチェックを実施する**（下記チェックリスト）
5. ユーザーレビュー後、指摘事項を反映する

ユーザーから「節構成は任せる」と明示された場合のみ、節構成案の確認をスキップして本文執筆に進んでよい。

## 8. 章別ガイドライン（要点）

各章で特に注意すべき点を以下に示します。執筆時はここを必ず確認してください。

### 第 1 章 はじめに
- Continue とは何か、なぜエアギャップ環境で使うのかを明確に提示
- 「対象読者」「前提条件」「読了後に到達できる状態」を冒頭で宣言

### 第 2 章 前提条件と環境確認
- VS Code バージョン、ローカル LLM の疎通確認（`curl` 等）を含める
- 「LLM 接続設定が完了している」とは何が満たされている状態かを明文化

### 第 3 章 VS Code 拡張のオフラインインストール
- Marketplace 経由ではなく VSIX による導入手順を中心に
- 社内配布リポジトリ・バージョン固定・更新運用に言及

### 第 4 章 テレメトリと外部通信の遮断確認 ★重点
- 監査担当者にエビデンスとして提出可能な粒度で記述
- 設定項目（`allowAnonymousTelemetry` 等）と確認方法（パケットキャプチャ、プロキシログ、開発者ツール）をセットで
- 「外部通信がないことを保証する」と断定する記述は避け、「一般的な検証手順」として記述
- タグ例: `telemetry`、`security`、`audit`

### 第 5 章 config.yaml の基本構造
- ファイルの場所、YAML 構文の基礎、ロール別モデル割り当ての考え方
- ローカル LLM エンドポイントの記述例を Ollama / vLLM / LM Studio で各 1 つ

### 第 6 章 Chat 機能の基本
- AI コーディング初心者向けに、ハルシネーション・回答検証の重要性を補足
- `@file` `@codebase` 等の基本コンテキスト指定を扱う

### 第 7 章 Autocomplete（タブ補完） ★重点
- 補完専用モデルと Chat モデルの違いを丁寧に解説
- レイテンシ／精度のチューニング指針を含める
- **ハンズオンを 1 つ以上含める**
- タグ例: `autocomplete`、`performance`

### 第 8 章 Edit 機能 ★重点
- Cmd+L / Cmd+I の操作フロー、インライン diff の扱い
- **リファクタリングのハンズオンを 1 つ含める**
- タグ例: `edit`、`refactoring`

### 第 9 章 Agent 機能 ★重点
- Chat との違い、ツール承認モデル、安全に使う観点を強調
- エアギャップ環境で利用可能／不可能なツールの整理
- タグ例: `agent`、`tools`

### 第 10 章 社内モデルとコンテキスト管理 ★重点
- `@docs` / `@codebase` の使い分けと運用
- ローカル Embedding モデルの設定
- カスタムコンテキストプロバイダの作成例
- タグ例: `context`、`embedding`、`indexing`

### 第 11 章 カスタマイズ ★重点
- 社内コーディング規約を Rules に落とす具体例
- **カスタムスラッシュコマンドのハンズオンを 1 つ含める**
- タグ例: `customization`、`rules`、`slash-commands`

### 第 12 章 MCP サーバの社内利用（発展）
- ローカルで動かせる MCP サーバの例（ファイルシステム、社内 Git 等）
- セキュリティ上の留意点
- タグ例: `mcp`、`advanced`

### 第 13 章 セキュリティと運用
- 機密コード取り扱い、ログ運用、配布・更新運用、教育
- タグ例: `security`、`operations`

### 第 14 章 トラブルシューティング
- ログの場所、代表的な症状と切り分け手順、品質チューニング
- タグ例: `troubleshooting`、`logs`
