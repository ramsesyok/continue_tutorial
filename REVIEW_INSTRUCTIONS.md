# 横断レビュー指示書

本ファイルは、複数章にまたがる整合性チェックを担当するエージェント（以下「レビュアー」）への指示書です。各章の執筆完了後、または定期レビューの実施時に本書のチェックリストに従って横断レビューを実施してください。

!!! tip "本書の位置付け"
    本書は執筆セッションとは別セッションで横断的な品質を確保するためのものです。執筆セッション内で実施するセルフチェック（`AGENT_INSTRUCTIONS.md` 6 章）とは別物です。

---

## 1. 前提

- レビューは**執筆セッションとは別セッション**で実施することを推奨します（先入観のないチェックを行うため）
- レビュー開始前に、以下を必ず読んでください
  - `AGENT_INSTRUCTIONS.md`（全体）
  - `PROGRESS.md`（現在の進捗状況）
- レビュー対象は `docs/` 配下のすべての Markdown ファイルです
- 重大度の基準は本書「6. 重大度の定義」を参照してください

---

## 2. 実施フロー

1. 本書（`REVIEW_INSTRUCTIONS.md`）を読む
2. `AGENT_INSTRUCTIONS.md` を読み、執筆ルールを確認する
3. `PROGRESS.md` を確認し、レビュー対象とする章を決める
   - 全章レビュー / 重点章のみ / 特定章のみ、いずれかを最初に決めてユーザーと合意する
4. 「3. チェックカテゴリ」の順序でカテゴリごとに検査する
5. 検査結果を `docs/_reviews/REVIEW_REPORT_YYYY-MM-DD.md` に保存する（`_reviews` ディレクトリがなければ作成する。MkDocs ナビには含めない想定）
6. `PROGRESS.md` の「4. 横断レビュー実施記録」に追記する
7. High の指摘事項は `PROGRESS.md` の「5. 指摘事項管理」にも転記し、対応漏れを防ぐ

---

## 3. チェックカテゴリ

優先度の高い順に並べています。**3.1 から 3.6 までは必須**で、3.7 以降は時間に応じて実施してください。

### 3.1 エアギャップ前提への適合（最重要）

- 外部 API への接続を前提とする手順が含まれていないか
- 「インターネットから直接ダウンロード」「`pip install <pkg>`」「`npm install <pkg>`」等の表現が、オフライン配布を前提とする書き方になっているか
- クラウド埋め込み・クラウドモデル等、外部依存の機能に言及する場合、「**本構成では使用しません**」と明記されているか
- サンプル内の URL / ホスト名が架空のプレースホルダ（`<your-internal-host>` 等）であるか
- 外部 URL が本文の手順中ではなく「参考リンク」セクションに分離されているか

検出ヒント:

```bash
# 怪しい表現の抽出
grep -nE "pip install|npm install|curl https?://" docs/*.md
grep -nE "ダウンロード|インターネット|オンライン" docs/*.md
```

### 3.2 Frontmatter

- すべての Markdown ファイルに Frontmatter があるか
- `title` / `type` / `tags` の 3 項目が揃っているか
- `type` の値が `chapter` / `appendix` / `index` / `meta` のいずれかであるか
- 共通タグ（`continue`, `vscode`, `on-premise`, `air-gap`）が含まれているか
- `title` の章番号が章一覧と一致しているか

検出ヒント:

```bash
# Frontmatter で始まっていないファイルを検出
for f in docs/*.md docs/appendix/*.md; do
  head -1 "$f" | grep -q '^---$' || echo "NO FRONTMATTER: $f"
done
```

### 3.3 ファイル名・配置

- ファイル名が `NN-slug.md` 形式であるか
- 章番号と内容が一致しているか（例: `07-autocomplete.md` が Autocomplete 章であるか）
- 付録は `docs/appendix/` 配下にあるか
- 画像は `docs/images/<章番号 2 桁>/` に配置されているか

### 3.4 リンクの整合性

- 章間の相対リンクが正しいファイル名を指しているか
- 存在しないファイルへのリンクがないか
- 「次の章へ」のリンク先が章一覧と整合しているか
- 外部 URL は本文中ではなく「参考リンク」セクションに集約されているか

検出ヒント:

```bash
# 内部リンクの抽出と存在確認
grep -oE "\[[^\]]+\]\(([^)]+\.md)\)" docs/*.md
```

### 3.5 章構成

- 章冒頭に「この章で学ぶこと」があるか
- 章末に「まとめ」と「次の章へ」があるか
- 重点章（4 / 7 / 8 / 9 / 10 / 11）に、`AGENT_INSTRUCTIONS.md` 8 章で指定されたハンズオンが含まれているか
- 第 4 章には監査エビデンスとして使える具体的な手順が含まれているか

### 3.6 用語・表記の統一

- `AGENT_INSTRUCTIONS.md` の「9. 用語の表記統一」表に違反していないか
- 文体（です・ます調）に揺れがないか
- 半角英数字と日本語の間にスペースが入っているか（例外: 行頭・句読点前後）

検出ヒント:

```bash
# 表記揺れの代表例
grep -nE "VSCode|vscode|Visual Studio Code" docs/*.md
grep -nE "エアーギャップ|オンプレ[^ミ]" docs/*.md
grep -nE "ローカルLLM|local LLM" docs/*.md  # 「ローカル LLM」に統一
```

### 3.7 コードブロック

- すべてのコードブロックに言語識別子が付いているか
- `config.yaml` の例が最小限で有効な構成か
- コード内に実在の社内情報・機密情報が含まれていないか

検出ヒント:

```bash
# 言語識別子なしのフェンスを検出（おおまかな目安）
grep -nE '^```$' docs/*.md
```

### 3.8 見出し階層

- 本文中で H1（`#`）が使われていないか（タイトルは Frontmatter で表現）
- 見出しレベルが飛んでいないか（`##` → `####` 等）

### 3.9 重複と抜け漏れ

- 他章で十分に説明されている内容を再度詳述していないか（必要な場合は該当章へのリンクで参照する形に）
- `AGENT_INSTRUCTIONS.md` の「8. 章別ガイドライン（要点）」に挙がっている項目が漏れていないか
- 章をまたぐ用語定義の重複（同じ用語の説明が複数章にある）

### 3.10 図表・画像

- 画像参照パスが相対パス（`images/<章番号>/...`）になっているか
- 代替テキストが意味のある内容になっているか（「画像」「figure」などの汎用語でないか）
- 画像内に機密情報が含まれていないか（スクリーンショットのファイルパス、内部ホスト名等）

## 4. レビューレポートの作成

レビュー結果は `docs/_reviews/REVIEW_REPORT_YYYY-MM-DD.md` として保存します。

### 4.1 Frontmatter

```yaml
---
title: "横断レビューレポート YYYY-MM-DD"
type: meta
tags:
  - meta
  - review
  - report
---
```

### 4.2 含めるべきセクション

1. **レビュー範囲**: 対象ファイル一覧、レビュアー（セッション識別子等）、レビュー実施日時
2. **サマリ**: 重大度別の件数（High / Medium / Low）と、特に重要な所見の 3〜5 行
3. **指摘事項**: 重大度ごとに分けて記述。各指摘には次の項目を含める
    - 対象ファイル（例: `docs/07-autocomplete.md`）と該当行
    - 違反しているチェック項目（例: `3.1 エアギャップ前提への適合`）
    - 現状の記述（短い引用）
    - 推奨対応
4. **全体所見**: 横断的に見えた傾向、改善のための提案

### 4.3 指摘事項の記述例

> **[High] エアギャップ前提への違反**
>
> - 対象: `docs/05-config-yaml-basics.md` 42 行目
> - チェック項目: 3.1 エアギャップ前提への適合
> - 現状の記述: 「`pip install transformers` で必要なライブラリをインストールします。」
> - 推奨対応: 社内ミラーからのオフライン配布前提に書き換える。例: 「あらかじめ社内パッケージミラーで配布された `transformers` を `pip install --no-index --find-links <ローカルパス> transformers` でインストールします。」

---

## 5. レビュー完了後の作業

1. `PROGRESS.md` の「4. 横断レビュー実施記録」に 1 行追加する（実施日、範囲、レポートへのリンク、件数、対応状況）
2. High に分類した指摘事項は `PROGRESS.md` の「5. 指摘事項管理」にも転記する
3. 担当章のステータスが「完了」になっている章に High 指摘が見つかった場合は、ステータスを「修正中」に戻し、メモ欄にレビュー指摘番号を記載する
4. ユーザーにサマリを報告する（指摘件数、High の概要、推奨対応の優先順位）

---

## 6. 重大度の定義

| 重大度 | 該当する例 | 対応の目安 |
| --- | --- | --- |
| **High** | エアギャップ前提への違反、Frontmatter 欠落、リンク切れ、機密情報の混入、重点章のハンズオン欠落 | リリース判定前に必ず修正 |
| **Medium** | 用語表記の揺れが多数、見出し階層の飛ばし、章構成の必須要素欠落（「まとめ」欠如 等）、章間の内容重複 | 次回更新までに修正 |
| **Low** | 軽微なタイポ、表記スタイルの細かい揺れ、画像の代替テキストの改善余地 | 余裕のあるタイミングで修正 |

High が 1 件でも残っている状態で、章のステータスを「完了」に戻さないでください。

---

## 7. 推奨レビュー周期

- **章単位**: 各章のセルフチェック完了直後に、その章だけを対象にした軽量レビュー
- **節目レビュー**: 重点章（4 / 7 / 8 / 9 / 10 / 11）すべてが「レビュー待ち」以上になった時点で、重点章のみの中間レビュー
- **全体レビュー**: すべての章が「レビュー待ち」以上になった時点で、全章対象の総合レビュー
- **リリース前最終レビュー**: 公開直前に High カテゴリのみを再確認

---

レビューの目的は「執筆者が見落としやすい横断的な不整合」を機械的に検出することです。指摘は具体的な箇所・推奨対応とセットで提示し、執筆セッションが受け取ってそのまま対応できる粒度にしてください。
