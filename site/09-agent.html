    第9章 Agent 機能を理解する

    この章で学ぶこと¶

    • Agent モードのねらいと、Chat / Edit との役割の違い
    • ツール承認モデル（自動承認 / 都度承認 / 拒否）の仕組みと安全な運用方針
    • エアギャップ環境で使えるツールと、避けるべきツールの切り分け
    • 小さなタスクを Agent に任せて完了まで見届けるハンズオン
    • Agent を業務で安全に使うためのガイドライン

    Agent 機能とは何か¶

    Chat / Edit との立ち位置¶

    ここまでの章で、コードについて質問する Chat（第 6 章）、コードを書き換える Edit（第 8 章）、コードを書くそばから補完する Autocomplete（第 7 章）を扱ってきました。Agent は、これらと並ぶ 4 つ目の機能であり、ひとことで言えば 「ツールを呼び出して自律的に作業する AI」 です。

    Chat は質問への回答を返すだけで、ファイルやコマンドを直接操作することはありません。Edit はユーザーが選択した範囲を 1 回だけ書き換えます。これに対して Agent は、ユーザーから受け取ったゴールを達成するために、ファイルを読み、コードを編集し、必要ならターミナルでコマンドを実行する、といった一連の作業を自分で判断しながら進めていきます。

    機能 入力 主な出力 ファイル操作 コマンド実行
    Chat 質問・依頼 回答テキスト しない しない
    Edit 選択範囲＋指示 1 回の差分提案 する（差分提案） しない
    Agent ゴール（依頼） ツール呼び出しの連鎖と最終応答 する（複数回） する（承認制）

    Agent が真価を発揮するのは、1 回の指示で 2 つ以上のステップを伴う作業です。「あるモジュールに関数を追加し、対応するテストも書き、テストを実行して通るまで直す」といった依頼を、人手による細かい操作なしに進められます。

    Agent の動作ループ¶

    Agent は、内部で次のサイクルを繰り返しながらゴールに近づきます。

    1. 思考: 現在の状況を踏まえ、次に何をすべきかを考える
    2. ツール呼び出し: ファイル読取、編集、コマンド実行などのツールを選び、引数を組み立てる
    3. 観測: ツールの実行結果（読んだファイルの中身、コマンドの標準出力など）を受け取る
    4. 判断: 結果を踏まえて、次のステップに進むか、ゴールに到達したかを判断する

    このループが目に見える形で動くため、Agent は単なる「自動化スクリプト」とは異なります。毎ステップで状況を再評価しながら、必要なら方針を変える という点が「自律実行」と呼ばれるゆえんです。

    Agent ＝ 全自動ではない

    Agent は自律的に判断しますが、エアギャップ環境での標準的な使い方では、ファイル編集やコマンド実行はユーザーの承認を必要とします。Agent が暴走してリポジトリを壊す、といった事態は、後述するツール承認モデルによって防がれます。

    Agent に向く仕事・向かない仕事¶

    すべての作業を Agent で行うのが効率的なわけではありません。次のような切り分けが目安になります。

    • Agent が向く: 「コードを読む → 直す → テストを走らせる → 失敗を直す」のように、複数の手順や試行錯誤が必要な作業
    • Edit が向く: 範囲が明確で 1 回の書き換えで済む作業（変数名のリネーム、関数のシグネチャ変更など）
    • Autocomplete が向く: 入力中のコードに対する短い補完
    • Chat が向く: ファイルを変更しない調査や、設計の相談、レビュー

    迷ったら、まず Chat で相談し、修正範囲が見えたら Edit、複数ステップを任せたいときに Agent、と段階的に切り替える運用が安全です。

    Agent モードを起動する¶

    モード切替と画面構成¶

    Continue のサイドパネルには、入力欄の近くに モード切替 のためのセレクタがあります。ここで Agent を選ぶと、入力欄が Agent モードに切り替わり、Chat とは別の挙動になります。

    操作手順は次のとおりです。

    1. VS Code 左側のアクティビティバーで Continue のアイコンをクリックし、サイドパネルを開く
    2. 入力欄上部または近傍のモード切替セレクタで Agent を選ぶ
    3. 入力欄に依頼を入力し、送信する

    Agent モードに切り替えると、サイドパネルに 使用可能なツールの一覧 が確認できます。ファイル読取、ファイル編集、ターミナル実行などのツールがリストアップされ、それぞれのツールが「自動承認」「都度承認」「拒否」のどの設定になっているかも見て取れます（詳細は後述）。

    モード切替のショートカット

    Chat と Agent の切り替えはサイドパネルから行うのが基本ですが、キーボードショートカットも割り当てられている場合があります。各バージョンのショートカットは、付録 B のキーボードショートカット表を参照してください。

    Agent に必要なモデル要件¶

    Agent モードでは、モデルが Tool Calling（関数呼び出し、ツール呼び出し） に対応している必要があります。これは「モデルが、定義済みのツールから 1 つを選び、引数を JSON で組み立てて返す」能力のことで、対応していないモデルでは Agent は動作しません（または不安定になります）。

    ローカル LLM では、次のような確認をおすすめします。

    • Ollama で動かす場合: モデルカード（ollama show <your-model-name> の出力など）に tools 対応の記載があるか、または公式ドキュメントで Tool Calling 対応とされているか
    • vLLM で動かす場合: 起動時に --enable-auto-tool-choice などのフラグを付与し、OpenAI 互換 API の Tool Calling を有効化しているか

    config.yaml 側は、Agent 用のチャットモデルとして roles に chat を含むモデルを 1 つ用意しておけば、通常はそのまま Agent からも利用できます。最小構成の例は次のとおりです。

    models:
  - name: internal-llm-chat
    provider: openai
    model: <your-model-name>
    apiBase: http://<your-internal-llm-host>:<port>/v1
    apiKey: <your-api-key-or-placeholder>
    roles:
      - chat

    Tool Calling 非対応モデルでの挙動

    Tool Calling に対応していないモデルを Agent モードで使うと、ツールを呼ばずに「テキストでこう変更してください」と提案するだけで止まったり、ツール呼び出しの JSON が壊れて失敗を繰り返したりします。Agent がうまく動かない場合は、まずモデルの Tool Calling 対応状況を疑ってください。

    ツール承認モデルを理解する¶

    3 つの承認モード¶

    Agent が外の世界に働きかけるすべての操作（ファイルを開く、ファイルを書き換える、シェルでコマンドを動かす、など）は、それぞれが 「ツール」 として定義されています。Continue は、各ツールに対して次の 3 段階の承認モードを設定できます。

    • 自動承認（Allow / Auto）: Agent がツールを呼ぶと、ユーザーへの確認なしに即座に実行されます。読取系の安全なツールに向きます。
    • 都度承認（Ask）: Agent がツールを呼ぼうとするたびに、ユーザーに承認ダイアログが表示されます。何をどう実行するかを見てから承認・却下を選びます。初学者と業務利用での既定はこれです。
    • 拒否（Deny）: そのツールは利用不可になります。Agent はそのツールを呼べません。

    設定は、Agent モードのサイドパネルに表示されるツール一覧の各ツールから個別に変更できます。プロジェクト全体のポリシーは config.yaml でも管理できる場合があります。

    承認ダイアログの読み方¶

    都度承認モードでは、Agent がツールを呼ぼうとすると、サイドパネル内に承認ダイアログが表示されます。ダイアログには、これから何が起こるかが具体的に示されます。

    • ファイル読取ツール: 読み取り対象のパス（例: src/task_sorter.py）
    • ファイル編集ツール: 編集対象のパスと、変更前後の差分（diff）
    • ターミナル実行ツール: これから実行されるシェルコマンドの全文（例: pytest tests/）

    ダイアログには通常 Accept（承認）と Reject（却下）のボタンがあります。承認する前に、次の 3 点を確認する習慣をつけてください。

    1. 対象: 編集や読取の対象パスが想定どおりか（特に .. で親ディレクトリへ出ようとしていないか）
    2. 内容: 編集差分やコマンドの中身が、依頼内容に対して妥当か
    3. 副作用: そのコマンドが取り消せる操作か、取り消せない操作か（rm や git push などは要注意）

    迷ったら却下

    承認ダイアログの内容に少しでも違和感を覚えたら、まず却下してください。Agent には「期待と違ったので、こうしてほしい」と伝えれば、別の手段で進めようとします。承認は「Yes」だけの一方通行ではなく、対話のきっかけです。

    推奨設定の例¶

    社内導入を想定した、現実的な推奨設定の例を示します。

    • 読取系: ファイル読取、コードベース検索などは 自動承認 にしてよい。Agent の応答が速くなり、開発体験が向上する
    • 編集系: ファイル編集は 都度承認 を既定にする。差分を必ず目で見てから承認する習慣を組織に定着させる
    • コマンド実行系: ターミナル実行は 都度承認 を既定にする。とくに rm、mv、git push、git reset --hard、データベースへの書き込みなど、取り消しが難しいコマンドは個別に 拒否 に設定するか、専用のラッパースクリプト経由でのみ実行する運用にする
    • 外部通信を伴うツール: 後述のとおり、エアギャップ環境では 拒否 に固定する

    config.yaml での記述例は次のとおりです（Continue のバージョンによりキー名が異なる場合があります。詳細は付録 A の config.yaml リファレンスを参照してください）。

    agent:
  tools:
    - name: read_file
      approval: auto
    - name: edit_file
      approval: ask
    - name: run_terminal
      approval: ask
    - name: web_fetch
      approval: deny

    エアギャップ環境で使えるツール・使えないツール¶

    組み込みツールの分類¶

    Continue が Agent 向けに提供する組み込みツールは、おおむね次の 5 カテゴリに整理できます。

    カテゴリ 主なツール 役割
    ファイル読取 ワークスペース内のファイル参照、ディレクトリ一覧 コードを読んで状況を把握する
    ファイル編集 ファイル新規作成、既存ファイルの書き換え コードを変更する
    コードベース検索 コードベース全体に対する検索、シンボル検索 関連箇所を探す
    ターミナル実行 シェルコマンドの実行 テスト、ビルド、リンタなどを動かす
    外部 API・Web 取得 Web ページ取得、外部検索 API 呼び出し インターネット上の情報を参照する

    このうち、上 4 カテゴリは原理的にローカル完結で動きますが、5 番目のカテゴリは外部ネットワークを前提とするため、エアギャップ環境では別扱いになります。

    エアギャップで利用可能なもの¶

    次のツールは、エアギャップ環境でもそのまま利用できます。

    • ファイル読取: ワークスペース内のファイル参照、ディレクトリ一覧
    • ファイル編集: ファイルの新規作成・上書き
    • コードベース検索: ローカルでインデックス化された検索（第 10 章で扱う Embedding を含む）
    • ターミナル実行: ローカルにインストール済みのコマンド（pytest、make、git など）

    ただし、ターミナル実行を介して 間接的に外部通信が発生する ケースには注意が必要です。代表例は次のとおりです。

    • pip install <package> のような、ネットからパッケージを取得しようとするコマンド
    • git clone https://... や git pull のような、外部リポジトリへのアクセス
    • curl / wget を含むコマンド

    これらは、エアギャップ環境では到達できず失敗するか、社内ミラーへの向き先が必要です。Agent がこうしたコマンドを実行しようとしたら、承認ダイアログで止めて Agent に方針変更を伝えてください（例: 「pip install は使わず、社内ミラーから事前に入れた前提で進めて」）。

    利用不可・要注意なツール¶

    次のツールは、エアギャップ環境では 本構成では使用しません。

    • Web ページ取得ツール: 任意の URL を取得しに行く動作のため、到達不可
    • 外部検索 API ツール: クラウド検索エンジン API を呼ぶ動作のため、到達不可
    • クラウド埋め込みモデル: 外部の埋め込み API を呼ぶ用途のもの（第 10 章で扱うローカル Embedding と置き換える）

    これらのツールは、Agent モードのツール一覧で 拒否 に設定してください。意図せず有効化されたままだと、Agent が「Web で検索する」「外部ドキュメントを取得する」といったツール呼び出しを試み、毎回失敗してループに陥る原因になります。

    保証ではなく確認

    本書はツールを拒否設定にすることを推奨しますが、これだけで「外部通信が起きないことを保証する」と断定はしません。実際の遮断は、第 4 章で扱ったネットワーク層・プロセス層の通信遮断と組み合わせて確認してください。

    ハンズオン：Agent に小さなタスクを任せる¶

    ここまでの内容を、実際に Agent を動かして体験します。題材は 「既存の Python モジュールに新しい関数を追加し、テストを書き、テストが通るまで直す」 という、Agent らしい複数ステップの作業です。

    題材の準備¶

    ワークスペース直下に、次の 2 つのファイルがある状態を出発点とします。手元で空のリポジトリを用意し、そのまま作成してください。

    src/task_sorter.py:

    """タスクの並び替えユーティリティ。"""

from typing import List, Dict, Any


def sort_by_due_date(tasks: List[Dict[str, Any]]) -> List[Dict[str, Any]]:
    """due_date キー（ISO 形式の日付文字列）で昇順に並べ替える。"""
    return sorted(tasks, key=lambda t: t["due_date"])

    tests/test_task_sorter.py:

    from src.task_sorter import sort_by_due_date


def test_sort_by_due_date():
    tasks = [
        {"id": 1, "due_date": "2026-06-10"},
        {"id": 2, "due_date": "2026-05-01"},
        {"id": 3, "due_date": "2026-07-15"},
    ]
    result = sort_by_due_date(tasks)
    assert [t["id"] for t in result] == [2, 1, 3]

    エアギャップ環境では、pytest が事前に社内配布物としてインストール済みであることを前提とします。pytest --version で実行できることを確認しておいてください。

    シナリオ¶

    Agent に次の依頼を出します。

    src/task_sorter.py に、priority キー（整数、小さいほど優先度が高い）で昇順にソートする sort_by_priority 関数を追加してください。あわせて tests/test_task_sorter.py に対応するテストケースを追加し、pytest tests/ を実行して通ることを確認してください。

    実行と承認の流れ¶

    依頼を送信すると、Agent は概ね次のような順序でツール呼び出しを試みます。承認ダイアログが表示されるたびに、ダイアログの中身を読み、Accept を押すか Reject を押すかを判断します。

    1. ファイル読取: src/task_sorter.py を読みに来る。読取は安全なので承認する（自動承認に設定済みなら確認は出ない）
    2. ファイル読取: tests/test_task_sorter.py を読みに来る。同様に承認する
    3. ファイル編集: src/task_sorter.py への差分が表示される。期待は sort_by_priority 関数の追加のみで、既存の sort_by_due_date を壊さないこと。差分を確認して承認する
    4. ファイル編集: tests/test_task_sorter.py への差分が表示される。新しいテストケースが追加されているだけで、既存テストが書き換えられていないことを確認して承認する
    5. ターミナル実行: pytest tests/ の実行が提案される。コマンド全文を確認して承認する
    6. 観測と判断: テストが通れば Agent はゴール到達と判断し、結果を要約して応答を返します

    各ステップで、次のような 読み方のコツ が役立ちます。

    • 差分の - 行を必ず見る: 追加（+）だけでなく、既存コードの削除（-）が含まれていないかを確認する。意図しない削除は事故のもと
    • コマンドの引数を端まで読む: pytest tests/ -x のような追加オプションが意図したものか確認する
    • 作業対象パスを確認する: 想定外のディレクトリ（/etc/ など、ワークスペース外）への参照がないか確認する

    失敗時の中断・やり直し¶

    ハンズオンでは、Agent が次のような「ハマり方」をすることがあります。これも Agent との付き合い方を学ぶ良い機会です。

    • テストが失敗する: 比較対象の順序を間違えるなど。Agent は失敗ログを読み取り、自分で src/task_sorter.py を修正しようとします。差分を見て妥当なら承認します
    • 想定外のファイルを編集しようとする: 例えば既存の sort_by_due_date を「ついでにリファクタする」差分を提案してきた場合は、Reject を押し、入力欄で「既存関数には触らないでください」と方針を伝えます
    • pip install を実行しようとする: エアギャップ環境では成功しません。Reject し、「環境には必要なパッケージは入っている前提で進めてください」と伝えます
    • 何度も同じ失敗を繰り返す: 入力欄で 「ここで止めてください」 と伝え、サイドパネルから会話を新規に開始（リセット）して、より具体的な指示でやり直します

    途中で止める

    Agent の応答中でも、サイドパネルの停止ボタン（多くの場合、四角の停止アイコン）で実行を中断できます。承認ダイアログを長く放置せず、止めるべきところで止めるのも、安全運用の一部です。

    Agent を安全に使うためのガイドライン¶

    ハンズオンを通して、Agent の挙動と承認ダイアログの読み方に触れました。最後に、業務利用で押さえておきたい運用上のガイドラインを整理します。

    破壊的操作への備え¶

    • Git 管理下で作業する: Agent に任せる作業は、常に Git でバージョン管理されたディレクトリで行います。直前にコミットしてから Agent を動かすと、失敗した場合に git checkout や git restore で容易に巻き戻せます
    • ブランチを切る: 既存の作業ブランチで直接動かさず、Agent 専用のブランチを切ってから依頼を出すと、影響範囲が局所化されます
    • 危険なコマンドはツール単位で拒否: rm -rf、git reset --hard、データベース削除コマンドなどは、ツールの承認モードを 拒否 に設定するか、専用のラッパースクリプト経由でのみ実行する運用にします
    • 対象ディレクトリを限定する: Agent のワークスペースを、機密度の低いリポジトリに限定して開きます。ホームディレクトリ全体を開いて Agent を起動するのは避けます

    機密ファイル・秘密情報の扱い¶

    • .env や鍵ファイルの読取要求は警戒する: Agent が「環境設定を確認するため .env を読みたい」と要求してきたら、承認ダイアログで一度立ち止まり、本当に必要かを判断します。多くの場合、依頼に対して .env の内容は不要です
    • 対象から除外する: .gitignore に近い形で、Continue が Agent のファイルアクセスから除外するパターンを設定できる場合があります。利用可能な設定は付録 A を参照してください
    • コードベース検索の結果も対象: ファイル読取だけでなく、コードベース検索の結果として秘密情報がモデルに渡る可能性もあります。秘密情報はそもそもリポジトリに含めない、という基本ルールを守ってください

    ログと履歴の確認¶

    • 会話履歴: Agent との会話は、Continue のサイドパネルから過去のセッションを呼び出して確認できます。何を依頼し、どんなツール呼び出しが行われたかを後から振り返れる状態にしておきます
    • ツール呼び出し履歴: 各ツールの呼び出しと結果は、サイドパネル内で展開して確認できます。事故が起きた際の原因究明に役立ちます
    • ログの保管: 業務利用では、組織のポリシーに合わせて、これらのログを一定期間保管する運用を検討してください。具体的なログの場所や保管設計は、第 13 章のセキュリティと運用で扱います

    Chat / Edit へのフォールバック判断¶

    Agent は便利ですが、万能ではありません。次のようなときは、Agent をいったん止め、Chat や Edit に切り替える判断が有効です。

    • Agent が同じ失敗を繰り返している: 一度 Chat に切り替え、エラーログを貼り付けて方針を相談する。方針が決まったら Edit で 1 箇所だけ手を入れる、という細かい運用に戻す
    • コンテキストが大きくなりすぎている: 会話が長くなると、Agent の判断精度は落ちていきます。区切りのよいところで新規セッションを開始する
    • 修正範囲が明確で 1 回で済む: わざわざ Agent に任せる必要がなく、Edit のほうが速く・安全に終わります

    「Agent で全部やる」ではなく、作業の性質に合わせて Chat / Edit / Agent を選び分ける ことが、エアギャップ環境で Continue を安全に使いこなす鍵になります。

    まとめ¶

    • Agent は「ツールを呼び出して自律的に作業する AI」であり、複数ステップを伴うタスクに向きます
    • Agent の操作はすべてツール呼び出しとして表現され、各ツールには自動承認 / 都度承認 / 拒否の 3 段階の承認モードを設定できます
    • エアギャップ環境では、ファイル読取・編集・コードベース検索・ローカルのターミナル実行は利用可能ですが、Web 取得や外部検索 API は本構成では使用しません
    • ハンズオンを通じて、関数追加・テスト追加・テスト実行までを Agent に任せ、承認ダイアログの読み方と中断・やり直しを体験しました
    • 業務利用では、Git とブランチでの保護、危険コマンドの拒否設定、機密ファイルへの警戒、Chat / Edit へのフォールバック判断が安全運用の柱になります
