## カスタマイズ機能の全体像

[第 5 章](05-config-yaml-basics.md) では `config.yaml` の基本構造を、[第 9 章](09-agent.md) では Agent 機能を学びました。Continue は初期状態でも十分に使えますが、社内固有の文脈——コーディング規約、レビュー観点、ドメイン用語——をあらかじめ仕込んでおくことで、LLM が生成するコードの品質を大きく底上げできます。

Continue が提供するカスタマイズの手段は、大きく 3 つに分かれます。

| 手段 | 設定場所 | 主な用途 | 適用タイミング |
| --- | --- | --- | --- |
| **Rules** | `config.yaml` の `rules` フィールド、または `.continuerules` ファイル | 社内コーディング規約など、常時適用したいシステムレベルの指示 | Chat・Edit・Agent のすべてのリクエストに自動挿入 |
| **カスタムプロンプト** | `config.yaml` の `prompts` フィールド | レビューや定型変換など、特定の操作を `/コマンド名` で即時呼び出し | ユーザーがコマンドを入力したときだけ実行 |
| **Prompt ファイル** | `.continue/prompts/` 配下の `.prompt` ファイル | チームで再利用する定型プロンプトを Git 管理 | ユーザーがファイル名で呼び出したときだけ実行 |

3 つの使い分けの基本方針は次のとおりです。**常に守ってほしいルール**は Rules に、**よく使うが毎回入力するのは面倒な操作**は `prompts` に、**チームで共有して Git で管理したいプロンプト**は Prompt ファイルに、というように役割を分担します。

!!! note "本章のエアギャップ前提"
    本章で扱う設定はすべてローカルファイルへの記述です。外部 API への接続や、インターネット上のサービスとの連携は一切含まれません。

## Rules で社内コーディング規約を伝える

### Rules とは何か

Rules（ルール）とは、Chat・Edit・Agent のあらゆるリクエストに対して、**LLM へのシステムメッセージとして自動的に挿入される指示文**のことです。システムメッセージとは、LLM が受け取るプロンプトの冒頭に置かれる「前提条件」の文章で、ユーザーが毎回入力しなくても常に効いている設定です。

Rules を活用すると、次のような効果が得られます。

- 「変数名は snake_case を使う」と設定しておくと、コード生成時に自動で従う
- 「コメントは日本語で書く」と設定しておくと、Chat や Edit の出力が常に日本語コメントになる
- 「外部ライブラリは社内承認済みのリストから選ぶ」と設定しておくと、未承認ライブラリの提案を抑制できる

なお、Rules はあくまで「LLM への指示文」であり、LLM がそれを完全に守ることを保証するものではありません。効果は LLM の能力と指示文の書き方に依存します。ルールへの適合率を高めたい場合は、後述する記述例を参考に、具体的かつ簡潔な指示文を心がけてください。

### Rules ファイルの作成と配置

Rules を設定する方法は 2 通りあります。1 つは `config.yaml` への直接記述、もう 1 つはプロジェクトルートに置く `.continuerules` ファイルへの記述です。

**方法 1：`config.yaml` に記述する（グローバル設定）**

`~/.continue/config.yaml`（ユーザーホームの Continue 設定ファイル）に `rules` フィールドを追加します。この方法は、すべてのプロジェクトに共通して適用したい規約に向いています。

```yaml
# ~/.continue/config.yaml の抜粋

rules:
  - 以下の規約に必ず従ってください。
  - 変数・関数名は snake_case を使用する
  - 1 つの関数は 1 つの責務のみを持つ
  - コメントは日本語で記述する
```

`rules` フィールドはシンプルな文字列のリストです。各要素が LLM へのシステム指示として渡されます。ファイルを参照する場合は `uses: file://パス` 形式も使えます。

```yaml
rules:
  - 回答は必ず日本語で行ってください。
  - uses: file://.continue/rules/coding-standards.md
```

**方法 2：`.continuerules` ファイルに記述する（プロジェクトローカル設定）**

プロジェクトのルートディレクトリに `.continuerules` というファイルを作成し、指示文を直接 Markdown で記述します。

```text
my-project/
├── .continuerules      ← ここに規約を書く
├── src/
└── ...
```

```markdown
# 社内コーディング規約

## 命名規則
- 変数・関数名は snake_case を使用すること
- クラス名は PascalCase を使用すること
- 定数は SCREAMING_SNAKE_CASE を使用すること

## コメント
- すべてのコメントは日本語で記述すること
```

`.continuerules` ファイルはプロジェクトリポジトリに含められるため、**チームメンバー全員が同じ規約を自動的に適用できる**というメリットがあります。個人の `config.yaml` に手動で設定を追加する必要がないため、新しいメンバーの環境セットアップが楽になります。

!!! tip "グローバルとローカルの使い分け"
    「すべてのプロジェクトに適用したい規約」は `config.yaml` のグローバル設定に、「このプロジェクト固有の規約」は `.continuerules` ファイルに書くと整理しやすくなります。両方が設定されている場合、Continue はどちらも LLM に渡します。

### 社内規約を Rules に落とす — 記述例

Rules を書くときに重要なのは、**「どう書くと LLM に正確に伝わるか」** という視点です。以下に、よくある社内規約の 3 つのカテゴリについて、不明確な書き方（NG 例）と、LLM に伝わりやすい書き方（OK 例）を対比して示します。

#### 命名規則

多くのチームで「命名規則が一番ぶれやすい」という声を聞きます。曖昧な指示は LLM も一貫して守れません。

snake_case（スネークケース）とはアンダースコアで単語をつなぐ形式（例: `user_name`）、PascalCase（パスカルケース）とは各単語の頭文字を大文字にする形式（例: `UserRepository`）、SCREAMING_SNAKE_CASE とはすべて大文字のスネークケース（例: `MAX_RETRY_COUNT`）を指します。

**NG 例**（曖昧すぎる）

```text
- 適切な命名規則を使ってください
```

**OK 例**（具体的・網羅的）

```text
## 命名規則
- 変数・関数名は snake_case を使用すること（例: user_name, get_user_list）
- クラス名は PascalCase を使用すること（例: UserRepository, OrderService）
- 定数は SCREAMING_SNAKE_CASE を使用すること（例: MAX_RETRY_COUNT）
- ブール値を返す関数名は is_ または has_ で始めること（例: is_valid, has_permission）
- 省略語は使わないこと（例: btn ではなく button、usr ではなく user）
```

#### エラーハンドリング

エラーハンドリングのパターンはプロジェクトやチームごとにかなり異なります。LLM はデフォルトで「一般的な」パターンを出力しますが、社内の方針を明示すれば従わせることができます。

**NG 例**（抽象的すぎる）

```text
- エラーは適切に処理してください
```

**OK 例**（パターンと理由を明記）

```text
## エラーハンドリング
- 例外は必ずキャッチし、素通りさせないこと
- except 節では、具体的な例外クラスを指定すること（except Exception: は禁止）
- エラー発生時は、ログを必ず出力した後に処理を継続または再 raise すること
- ユーザー向けエラーメッセージには、内部のスタックトレースや技術的詳細を含めないこと
```

Rules のコードスニペット例は、Markdown のコードブロックとして記述します。

```python
## エラーハンドリングの例（Python）
try:
    result = call_internal_api(endpoint, params)
except RequestTimeoutError as e:
    logger.error("API タイムアウト: endpoint=%s, error=%s", endpoint, e)
    raise ServiceUnavailableError("サービスが一時的に利用できません") from e
```

コードスニペットを含めると、LLM がパターンを正確に再現しやすくなります。

#### ログ出力フォーマット

ログフォーマットは、後の運用・監視に直結する重要な規約です。構造化ログ（JSON 形式等）を採用している組織では、特に明示的な指示が効果的です。

**NG 例**（形式が不明確）

```text
- ログは構造化して出力してください
```

**OK 例**（具体的なフォーマットと必須フィールドを指定）

```text
## ログ出力規約
- ログは必ず構造化ログ（JSON 形式）で出力すること
- ログレベルは DEBUG / INFO / WARNING / ERROR / CRITICAL の 5 段階を使用する
- INFO 以上のログには以下のフィールドを必ず含めること
  - timestamp: ISO 8601 形式（例: 2026-01-15T10:30:00+09:00）
  - level: ログレベル
  - message: 人間が読める説明文（日本語可）
  - module: ログを出力したモジュール名
- 個人情報・認証情報（パスワード、トークン等）は絶対にログに出力しないこと
```

!!! warning "Rules は万能ではない"
    Rules に書いた内容は LLM の応答に強く影響しますが、**100% 守られることは保証されません**。特に生成コードが長くなるほど、ルールの適用漏れが起きやすくなります。重要な規約は CI のリンター・フォーマッターでも機械的に検証する構成を合わせて整備することを推奨します。

## カスタムプロンプトを作る

### `prompts` とは

`prompts`（プロンプト）とは、`config.yaml` に定義する `/コマンド名` 形式のショートカットプロンプトです。Chat パネルの入力欄に `/` から始まる短い文字列を打つことで呼び出せます。

`prompts` を使うと、**チームが繰り返し使いたいプロンプトをコマンド 1 つで呼び出せる**ようになります。

カスタムプロンプトが向いている用途の例を以下に示します。PR（Pull Request）とは、Git リポジトリにおいてコードの変更をチームに提案するための仕組みです。

- `/myreview` — 社内の PR レビュー観点でコードをチェックさせる
- `/docstring` — 社内フォーマットに沿ったドキュメントコメントを生成させる
- `/unittest` — 社内のテストフレームワーク（pytest 等）に合わせたユニットテストを生成させる
- `/changelog` — 変更差分からリリースノートの草案を作成させる

### ハンズオン：社内レビュー観点チェッカーを作る

ここでは、選択したコードに対して社内の PR レビュー観点を自動チェックする `/myreview` コマンドを作成します。

#### ステップ 1：`config.yaml` に `prompts` を追加する

`~/.continue/config.yaml` を開き、`prompts` フィールドを追加します。既に他の設定がある場合は、同じファイルに追記してください。

```yaml
# ~/.continue/config.yaml の抜粋

prompts:
  - name: myreview
    description: "社内 PR レビュー観点でコードをチェックする"
    prompt: |
      以下のコードを、社内の PR レビュー観点から確認し、問題点と改善案を日本語で報告してください。

      ## レビュー観点
      1. **セキュリティ**: SQL インジェクション（悪意ある SQL 文を埋め込む攻撃）・パストラバーサル（許可されたディレクトリ外へのファイルアクセス）・認証情報のハードコード等がないか
      2. **パフォーマンス**: N+1 問題・無駄なループ・不必要な I/O 呼び出しがないか
      3. **命名規則**: 変数・関数・クラス名が社内規約（snake_case / PascalCase）に沿っているか
      4. **エラーハンドリング**: 例外が適切にキャッチ・ログ出力されているか
      5. **可読性**: 関数が単一の責務に収まっているか・マジックナンバーがないか

      ## 報告フォーマット
      各観点について「問題なし」または「【問題あり】<具体的な問題点> → 改善案: <提案>」の形式で出力してください。
```

`name` がコマンド名（スラッシュの後ろに入力する文字列）です。`description` は `/` を入力したときにサジェストとして表示されます。`prompt` が LLM に送られる実際の指示文です。

#### ステップ 2：VS Code を再起動して設定を反映する

`config.yaml` を保存したら、VS Code を再起動するか、コマンドパレット（`Ctrl+Shift+P` または `Cmd+Shift+P`）で「Continue: Reload Config」を実行して設定を再読み込みします。

!!! note "設定が反映されない場合"
    `config.yaml` の YAML 構文が誤っている場合、コマンドが表示されないことがあります。VS Code の Continue パネル下部にエラーメッセージが表示されていないか確認してください。

#### ステップ 3：コマンドを使ってみる

1. VS Code でレビューしたいコードファイルを開く
2. レビュー対象の関数やクラスをマウスで選択する
3. Continue の Chat パネルを開く（`Ctrl+L` または `Cmd+L`）
4. 選択範囲が自動的にコンテキストに追加されていることを確認する
5. 入力欄に `/myreview` と入力し、サジェストから選択して Enter を押す

LLM が 5 つの観点それぞれについて評価を返します。問題が見つかった場合は、Chat パネルでそのまま「3 番の問題を修正してください」のように続けて会話できます。

N+1 問題とは、1 件のリクエストに対して N 件の追加クエリが発生してしまうパフォーマンス上の問題です。たとえばユーザー一覧を取得した後、各ユーザーの詳細を個別にループで取得するような実装がこれにあたります。

**実行例（出力イメージ）**

```text
## レビュー結果

1. セキュリティ: 問題なし

2. パフォーマンス:
   【問題あり】 get_users() 内で、ループの中に DB クエリが含まれており N+1 問題が発生しています。
   → 改善案: ユーザーリストを一括取得してからメモリ上でフィルタリングするか、JOIN クエリで
     まとめて取得する構成に変更してください。

3. 命名規則: 問題なし

4. エラーハンドリング:
   【問題あり】 15 行目の except: が素通りしており、例外が握り潰されています。
   → 改善案: except SpecificError as e: の形式に変更し、logger.error() で記録してください。

5. 可読性: 問題なし
```

#### カスタマイズのヒント

`/myreview` の `prompt` は出発点に過ぎません。チームで使い続けながら育てていくことが大切です。

- **観点を追加する**: 「テストが書きやすい構造か」「ドキュメントコメントが付いているか」など、チームの関心に応じて観点を増やせます
- **言語・フレームワークを限定する**: Python 向け・Java 向けなど複数バリエーションを `name: myreview-python` `name: myreview-java` のように作り分けることもできます
- **段階的に改善する**: 最初は「出力が期待と違う」という部分から prompt を少しずつ修正し、チームメンバーにも試してもらいながら精度を上げていきましょう

!!! tip "プロンプトの個数に制限はない"
    `prompts` リストに複数のエントリを並べて定義できます。よく使う操作をどんどんコマンド化して、個人やチームの作業を効率化してください。

## Prompt ファイルでプロンプトを再利用する

### Prompt ファイルとは

Prompt ファイルは、`.continue/prompts/` ディレクトリに置く `.prompt` 拡張子のテキストファイルです。スラッシュコマンドと似ていますが、以下の点が異なります。

| 比較項目 | スラッシュコマンド | Prompt ファイル |
| --- | --- | --- |
| 定義場所 | `config.yaml` の `slashCommands` | `.continue/prompts/` 配下のファイル |
| Git 管理 | `config.yaml` ごと管理（個人設定と混在しやすい） | ファイル単位で管理（プロジェクトリポジトリに含めやすい） |
| パラメータ化 | プロンプト文字列に変数を埋め込みにくい | `{{{ input }}}` 等の変数が使える |
| 向いている用途 | 個人・チームの汎用ショートカット | プロジェクト固有の定型プロンプトのチーム共有 |

Prompt ファイルは特に、「このプロジェクトに参加したメンバー全員が同じプロンプトを使えるようにしたい」という場面で有効です。ファイルをリポジトリに含めておけば、クローンするだけで使える状態になります。

### Prompt ファイルの作成例

#### 例 1：テスト雛形生成プロンプト

`.continue/prompts/gen-unittest.prompt` というファイルを作成します。

```text
name: テストコード生成
description: 選択した関数に対して pytest のユニットテスト雛形を生成する
---
以下の関数に対して pytest を使ったユニットテストのコードを生成してください。

## 要件
- テストファイルの命名は test_<モジュール名>.py 形式にすること
- 正常系・異常系・境界値の 3 パターンを最低限カバーすること
- 各テスト関数には docstring で「何をテストしているか」を日本語で説明すること
- 外部依存（DB・API 等）は unittest.mock でモック化すること

## 対象コード
{{{ input }}}
```

`---` より前が Frontmatter（メタ情報）、後ろが実際のプロンプト本文です。`{{{ input }}}` の部分には、Chat パネルで選択された文字列が自動的に挿入されます。

#### 例 2：ドキュメントコメント追加プロンプト

```text
name: ドキュメントコメント追加
description: 社内フォーマットに沿ったドキュメントコメントを追加する
---
以下のコードに、社内フォーマットに沿ったドキュメントコメントを追加してください。

## 社内ドキュメントコメント規約
- Python の場合は Google スタイルの docstring を使用する
- Args / Returns / Raises セクションを必要に応じて記述する
- 説明文は日本語で記述する
- 「何をするか」（What）だけでなく「なぜそうするか」（Why）も可能な範囲で記述する

## 対象コード
{{{ input }}}
```

#### Prompt ファイルの呼び出し方

Prompt ファイルを作成して Continue が認識すると、Chat パネルの入力欄から `/gen-unittest`（ファイル名の `.prompt` を除いた部分）でコマンドとして呼び出せます。スラッシュコマンドと同じ操作感で使えます。

!!! note "`.continue/prompts/` ディレクトリが存在しない場合"
    プロジェクトルートに `.continue/` ディレクトリがなければ手動で作成してください。Continue は起動時にこのディレクトリを自動探索します。

## チームへの展開と運用

### 設定ファイルを Git で共有する

カスタマイズ設定をチームで共有する最も確実な方法は、**プロジェクトリポジトリに設定ファイルを含めること**です。

共有に適したファイルと配置を以下に示します。

```text
my-project/
├── .continuerules           # プロジェクト固有の Rules（全メンバーに自動適用）
├── .continue/
│   └── prompts/
│       ├── gen-unittest.prompt   # テスト生成プロンプト
│       └── add-docstring.prompt  # ドキュメントコメントプロンプト
├── src/
└── ...
```

この構成のメリットは 3 つあります。まず、新しいメンバーがリポジトリをクローンした時点で、すぐに同じ設定で作業を開始できます。次に、プロンプトや規約の変更が Git の差分として追えるため、「いつ・なぜ変えたか」が残ります。そして、プルリクエスト経由でプロンプトの改善を提案・レビューできます。

**グローバル設定との分離**

個人ごとに異なる設定（使用するモデルのエンドポイントや認証情報など）は `~/.continue/config.yaml`（グローバル設定）に置き、**チームで統一すべき設定だけをリポジトリに含める**という原則を守ってください。認証情報を誤ってリポジトリに含めないよう、`.gitignore` の管理も合わせて確認しましょう。

```gitignore
# .gitignore への追記例
# 個人の認証情報を含む可能性があるファイルは除外
.continue/config.local.yaml
```

!!! warning "`.continuerules` には機密情報を書かない"
    `.continuerules` はリポジトリに含めることを前提としています。社内 API のエンドポイント URL・認証トークン・個人を特定できる情報などは絶対に記述しないでください。プレースホルダ（例: `<your-internal-api-endpoint>`）を使い、実際の値は別の手段で管理してください。

### 更新・メンテナンスの運用ポイント

Rules や Prompt ファイルは、一度作ったら終わりではありません。コードベースの成長や規約の変化に合わせて継続的に見直す必要があります。

**定期的な見直しサイクル**

スプリントレトロスペクティブや四半期ごとの設計レビューなど、チームが定期的に集まるタイミングに「Continue 設定の見直し」を議題として組み込むと管理しやすくなります。「最近このコマンドを使っていない」「この規約は LLM が守れていない」といった気づきをその場で収集し、プロンプトの改善に反映させます。

**フィードバックの集め方**

日常的に使いながら感じた「この出力は期待と違う」という感覚を、その都度 Prompt ファイルや Rules に反映させることが質の向上につながります。チームの Slack チャンネルや社内 Wiki に「Continue カスタマイズ改善ログ」のような場を設けておくと、フィードバックが蓄積しやすくなります。

**エアギャップ環境でのファイル配布**

Prompt ファイルや `.continuerules` の更新は、社内 Git リポジトリ経由で `git pull` するだけで全メンバーに配布できます。インターネット接続がなくても、社内 Git サーバ（GitLab オンプレミス、Gitea 等）があれば同様の運用が可能です。

```bash
# チームメンバーは通常の git pull で最新の設定を取得できる
cd /path/to/my-project
git pull
# → .continuerules や .continue/prompts/ の変更が自動的に反映される
```

## まとめ

- **Rules** を使うと、社内コーディング規約（命名規則・エラーハンドリング・ログフォーマット等）を LLM に常時適用できる。`config.yaml` でグローバルに設定するか、`.continuerules` ファイルでプロジェクトローカルに設定する
- Rules は単純な文字列リストで記述し、`uses: file://パス` でファイル参照も可能。「具体的・網羅的・パターン付き」で書くほど LLM への伝達精度が上がる。ただし CI のリンターと組み合わせて機械的な検証も行うことが望ましい
- **カスタムプロンプト**を `config.yaml` の `prompts` に定義すると、繰り返すプロンプト操作を `/コマンド名` 1 つで実行できる。社内レビュー観点チェッカー `/myreview` はその代表的な活用例
- **Prompt ファイル**は `.continue/prompts/` に置く `.prompt` ファイルで、パラメータ化と Git 管理がしやすいチーム共有向けの手段
- チームへの展開は「プロジェクトリポジトリに含めて `git pull` で配布」が最もシンプルかつエアギャップ環境でも機能する運用方法
