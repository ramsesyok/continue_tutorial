## この章で学ぶこと

- <glossary:Chat> パネルの開き方と、メッセージ送受信の基本的な操作フローを理解する
- `@file` / `@codebase` 等のコンテキスト指定を使い、より精度の高い回答を引き出す方法を習得する
- <glossary:ハルシネーション>（誤情報の生成）とは何かを理解し、AI の回答を正しく検証する習慣を身につける
- Chat・<glossary:Edit>・<glossary:Agent> の 3 機能の使い分け基準を把握し、次章以降の学習につなげる

## Chat パネルを開く

### パネルの場所と起動方法

VS Code を開いた状態で、画面左サイドバーに Continue のアイコン（チャット吹き出しのようなマーク）が表示されています。このアイコンをクリックすると、Continue の Chat パネルがサイドバーに展開されます。

キーボードショートカットでも開くことができます。

| 操作 | Windows / Linux | macOS |
| --- | --- | --- |
| Chat パネルを開く / 閉じる | `Ctrl+L` | `Cmd+L` |

!!! tip "パネルが見当たらない場合"
    アイコンが見当たらないときは、VS Code のメニューから **[表示] → [<glossary:拡張機能>]** を開き、Continue の拡張機能が有効になっているか確認してください。有効になっていれば、サイドバー上部の「…」メニューからアイコンの表示切り替えができます。

### 画面の構成

Chat パネルを開くと、次の 3 つのエリアが目に入ります。

- **会話エリア**（中央の大部分）: AI とのやり取りが時系列で表示される領域です
- **入力欄**（パネル下部）: 質問やコードを入力してメッセージを送る欄です。`Enter` キーで送信、`Shift+Enter` で改行します
- **モデル選択**（入力欄の右端）: 使用するモデルを切り替えられます。第 5 章で設定した `config.yaml` の内容がここに反映されます

## 最初の質問をしてみよう

### 簡単な質問から試す

はじめて Chat を使う場合は、身近なコードに関する簡単な質問から始めてみましょう。たとえば次のようなメッセージを入力して `Enter` を押してみてください。

```text
Python でリストの重複を除去する方法を教えてください。
```

しばらくすると、AI からの回答が会話エリアに表示されます。回答にはコードブロックが含まれることが多く、コードブロックの右上にあるコピーボタンでクリップボードにコピーできます。

### 回答が届くまでの仕組み

入力したメッセージは、Continue を経由してローカル <glossary:LLM> へ送信されます。LLM はその内容をもとに回答を生成し、Continue を通じて VS Code の画面に表示されます。ネットワークの外へはデータが出ないため、社内のコードを含む質問をしても問題ありません。

!!! note "回答が表示されるまでの時間"
    回答生成にかかる時間は、使用しているローカル LLM のモデルサイズや、実行しているマシンのスペックによって異なります。数秒で返ってくる場合もあれば、数十秒かかる場合もあります。回答が遅いと感じる場合は、第 14 章のトラブルシューティングを参照してください。

## より良い回答を引き出す質問の書き方

### 質問の具体性が回答の質を左右する

AI コーディングアシスタントは、質問が曖昧であるほど的外れな回答を返しやすくなります。同じ「バグを直したい」という目的でも、次の 2 つの質問では回答の有用性が大きく変わります。

**曖昧な質問（避けるべき例）:**
```text
バグを直してください。
```

**具体的な質問（推奨する例）:**
```text
以下の Python 関数で、空のリストを渡したときに IndexError が発生します。
原因を特定し、修正例を示してください。

def get_first(items):
    return items[0]
```

具体的な質問には、次の 3 つの要素を含めると効果的です。

1. **言語・フレームワークの明記**: 「Python で」「FastAPI を使って」のように、使用技術を明示する
2. **何が問題か・何をしたいかの具体的な説明**: エラーメッセージ、期待する動作、実際の動作を整理して伝える
3. **期待する出力形式の指定**: 「コード例を示してください」「箇条書きで整理してください」のように、回答の形式を指定する

### 追加で質問して会話を深める

一度の回答で満足できなかった場合は、そのまま続けて質問できます。会話の文脈は引き継がれているため、「もう少し詳しく教えてください」「別のアプローチもありますか？」のように追加の質問を重ねて、回答を掘り下げることができます。

## コンテキストを指定する

Chat の強みの一つは、現在作業しているコードをそのまま参照させながら質問できることです。Continue では「`@`（アットマーク）記号」を使ってコンテキスト（参照対象）を明示的に指定できます。

コンテキストを指定することで、AI は「あなたのプロジェクト固有の事情」を踏まえた回答を生成しやすくなります。

### `@file` ― 特定ファイルを参照させる

`@file` を使うと、特定のソースファイルの内容を AI に参照させることができます。

**使い方:** 入力欄に `@file` と入力すると、プロジェクト内のファイルが一覧表示されます。対象ファイルを選択すると、そのファイルがコンテキストに追加されます。

```text
@file src/auth/login.py

この関数が返している JWT トークンの有効期限ロジックを説明してください。
```

`@file` は次のような場面で特に役立ちます。

- 特定のファイルにあるバグの原因を調べたいとき
- 既存の関数の仕様を説明してもらいたいとき
- ファイルの構成を理解したうえで改修案を提案してもらいたいとき

!!! tip "複数ファイルの指定"
    `@file` は 1 つのメッセージに複数指定できます。たとえば `@file src/models/user.py` と `@file src/api/users.py` を同時に指定し、「この 2 つのファイルの依存関係を説明してください」という形で質問することも可能です。

### `@codebase` ― リポジトリ全体を対象にする

`@codebase` を使うと、プロジェクト全体のコードを検索・参照した上で回答を生成するよう AI に指示できます。

**使い方:** 入力欄に `@codebase` と入力するだけです。

```text
@codebase

このリポジトリで認証処理を担っているファイルはどれですか？
```

`@codebase` は次のような場面に向いています。

- リポジトリ全体の構造を把握したいとき
- 特定の処理がどのファイルに実装されているか調べたいとき
- 横断的な影響範囲を確認したいとき

!!! warning "コードベースのインデックス"
    `@codebase` を使うには、事前にリポジトリの<glossary:インデックス>（検索用の索引）が作成されている必要があります。初回はインデックス作成に時間がかかる場合があります。詳細な設定方法は第 10 章で解説します。また、インデックス作成に使用する <glossary:Embeddingモデル>はローカルで動作するため、コードが外部へ送信されることはありません。

### その他の基本コンテキスト修飾子

`@file` と `@codebase` 以外にも、よく使われる修飾子があります。

| 修飾子 | 参照対象 | 使いどころの例 |
| --- | --- | --- |
| `@terminal` | VS Code のターミナル出力 | エラーログやコマンド結果をそのまま質問に含めたいとき |
| `@problems` | VS Code の「問題」パネルの警告・エラー一覧 | 現在の lint エラーや型エラーを一括で確認・修正したいとき |
| `@open` | 現在 VS Code で開いているすべてのタブ | 複数ファイルを横断した質問をしたいとき |

!!! note "コンテキストの選び方"
    コンテキストは「必要最小限」を意識して選ぶことをおすすめします。関係のないファイルを大量に含めると、AI が不要な情報に引きずられて回答の精度が下がることがあります。まずは `@file` で関連ファイルを絞り込み、それでも答えが得られない場合に `@codebase` を試すという順序が効果的です。

## ハルシネーションと回答の検証

### ハルシネーションとは何か

AI が自信満々に誤った情報を返す現象を「ハルシネーション（幻覚）」と呼びます。これは LLM の構造的な特性であり、どれだけ高性能なモデルでも避けられません。

ハルシネーションが起きる典型的な場面としては、次のようなケースがあります。

- 存在しない関数名や API を「あるもの」として説明する
- 自信ありげに誤ったコードを提示する
- 古いバージョンの仕様をそのまま説明する（バージョンが更新されているにもかかわらず）

特に<glossary:オンプレミス>環境では、社内独自のフレームワークや内製ライブラリについてモデルが学習していないため、「そのライブラリは存在しない」と判断してしまったり、類似した別のものと混同して説明してしまったりすることがあります。

### 回答を検証する習慣をつける

AI の回答は「提案」として受け取り、必ず自分の目で確認することが重要です。具体的には次の点を意識してください。

1. **コードは動かして確認する**: 提示されたコードをそのまま本番環境に取り込まず、まずローカルのテスト環境で動作を確認する
2. **根拠を求める**: 「なぜそのコードで動作するのですか？」と追加質問し、説明が論理的かどうかを確認する
3. **社内ドキュメントや公式仕様と照合する**: 内製ライブラリや社内フレームワークに関する回答は、社内の仕様書やソースコードと必ず突き合わせる

!!! warning "ハルシネーションはなくせない"
    設定やモデルを工夫してもハルシネーションをゼロにすることはできません。「AI が正しいことを前提とする」のではなく「AI は間違えることがある」という前提で使うことが、安全・安心な活用の第一歩です。AI の回答の精度に不安を感じたら、別のモデルで同じ質問を試してみる、または社内の別メンバーにレビューを依頼することも有効です。

### コンテキストを渡すとハルシネーションは減る

ハルシネーションを減らす有効な方法の一つが、前節で説明したコンテキストの活用です。社内ライブラリのソースファイルを `@file` で指定しながら質問すると、AI は実際のコードを参照して回答を生成できるため、誤りが混入しにくくなります。

質問の精度を上げたい場合は、次の順序を意識してみてください。

1. まず質問だけ送ってみる
2. 回答が的外れな場合は `@file` で関連ファイルを指定して再質問する
3. それでも改善しない場合は `@codebase` に広げる

## 会話の管理 ― 新規チャットとコンテキストリセット

### 会話を引き継ぐ場合と始め直す場合

Continue の Chat では、会話が続いている間は前のやり取りが「文脈」として保持されます。同じトピックについて追加質問するときは、そのまま続けて質問するのが便利です。

一方、別のトピックに移りたいときや、前の会話の流れが邪魔になっていると感じるときは、**新しい会話（New Chat）を始める**ことをおすすめします。

新しい会話を始めるには、Chat パネル上部の「**+**」ボタンをクリックするか、次のショートカットを使います。

| 操作 | Windows / Linux | macOS |
| --- | --- | --- |
| 新しい会話を開始する | `Ctrl+Shift+L` | `Cmd+Shift+L` |

### コンテキストウィンドウの有限性

LLM には「<glossary:コンテキストウィンドウ>」と呼ばれる、一度に処理できるテキスト量の上限があります。会話が長くなりすぎると、古いやり取りが次第にコンテキストから押し出され、回答の一貫性が低下することがあります。

長い会話でこれを感じた場合は、新しい会話を始めて質問を整理し直すのが効果的です。

!!! tip "過去の会話を参照する"
    Continue は過去の会話履歴を保存しています。Chat パネル上部の時計アイコン（履歴ボタン）をクリックすると、以前のセッションのやり取りを確認できます。

## Chat / Edit / Agent の使い分け

Continue には Chat の他に、Edit（コードの書き換え）と Agent（自律的なタスク実行）という機能もあります。どれも AI を活用する機能ですが、目的によって使い分けることで、より効率よく作業できます。

| 機能 | 主な用途 | 典型的な操作 |
| --- | --- | --- |
| **Chat** | 質問・調査・説明を求める | 「この関数の意味を教えて」「この設計は適切ですか？」 |
| **Edit** | 選択したコードを書き換える | 「この関数をリファクタリングして」「バグを修正して」 |
| **Agent** | 複数ステップの作業を自動で進める | 「テストを書いて実行し、結果を報告して」 |

本章では Chat の基本操作を学びました。Edit の使い方は第 8 章で、Agent の仕組みと安全な使い方は第 9 章で詳しく説明します。

!!! note "まずは Chat から始める"
    AI コーディングが初めての方は、まず Chat での「対話」に慣れることをおすすめします。Chat で質問しながら理解を深め、コードの変更に自信が持てるようになったら Edit を試す、という段階的なアプローチが安全です。

## まとめ

- Chat パネルは `Ctrl+L`（macOS: `Cmd+L`）で開き、入力欄にメッセージを入力して `Enter` で送信する
- 言語・フレームワーク・期待する出力形式を明記した具体的な質問が、回答の精度を高める
- `@file` で特定ファイルを、`@codebase` でリポジトリ全体をコンテキストに指定できる。コンテキストは「必要最小限」を意識して選ぶ
- AI はハルシネーション（誤情報の生成）を起こすことがある。生成されたコードは必ず動作確認し、社内仕様書やソースコードと照合する習慣をつける
- 別のトピックに移るときは新しい会話を始めて、コンテキストを整理する
